# Faster Debugging with ECS Workloads

## Problem

Debugging the `mw-agent` running inside an ECS task is painful. Reproducing issues locally is often impossible because the agent relies on the ECS metadata endpoint and container-level metrics that only exist inside a real ECS environment. The traditional cycle of "add a log line, build, push, redeploy, wait, read CloudWatch logs" is slow and makes root-cause analysis tedious — especially for issues in the AWS ECS container metrics receiver or related internal packages.

## What These Changes Add

Four new files work together to give you a **live remote debugging session** against the agent running on ECS, directly from your IDE.

### 1. `Dockerfiles/DockerfileDebug` — Debug-Ready Container Image

A multi-stage Dockerfile that produces a container suitable for remote debugging:

- **Build stage** — compiles the agent with `"-gcflags "all=-N -l"` to disable optimizations and inlining so that Delve can map execution back to source accurately. It also copies in `local-deps/` so you can test local modifications to upstream collector-contrib packages without publishing them.
- **Runtime stage** — installs [Delve](https://github.com/go-delve/delve) (`dlv`), exposes port **2345**, and starts the agent under Delve in headless mode with `--accept-multiclient`.

This means the container boots up and immediately waits for a debugger to attach.

### 2. `ecs-build.sh` — One-Command Build & Push

Automates the entire build-push cycle:

1. Copies the three packages you are most likely to modify (`awsecscontainermetricsreceiver`, `ecsutil`, `internal/common`) from your local `opentelemetry-collector-contrib` checkout into `local-deps/`.
2. Builds a linux/amd64 image using `DockerfileDebug`.
3. Tags it with a timestamp (`dev-<epoch>`) and pushes to `ghcr.io/middleware-labs/mw-host-agent`.
4. Prints the exact `aws ecs update-service` command to trigger a redeployment.

Running a single `./ecs-build.sh` replaces what used to be several manual steps.

### 3. `update-debug-host.sh` — Auto-Discover the Debug Target

ECS tasks can land on any EC2 instance in the cluster, and the instance's public IP changes across deployments. This script:

1. Queries the ECS API to find the running task for `mw-agent-daemon-service`.
2. Resolves the task → container instance → EC2 instance → public IP chain.
3. Updates `.vscode/launch.json` with the discovered IP.

After running `./update-debug-host.sh`, your IDE debug configuration is already pointing at the right host — no manual copy-pasting.

### 4. `.vscode/launch.json` — IDE Debug Configuration

A pre-configured Cursor/VS Code launch configuration that attaches to the remote Delve server:

- **Mode:** `remote` attach over port `2345`.
- **Path substitution:** maps the local `${workspaceFolder}` to `/go` inside the container, so breakpoints set on local source files resolve correctly.
- **`stopOnEntry: true`** — pauses the agent at the first instruction so you can set breakpoints before any startup logic runs.

### 5. `.gitignore` — Keeps `local-deps/` Out of Source Control

The vendored `local-deps/` directory is generated by `ecs-build.sh` and should not be committed. The updated `.gitignore` ensures it stays out of diffs and PRs.

## End-to-End Workflow

```text
# 1. Make changes to mw-agent or the collector-contrib packages locally.

# 2. Build, push, and get the image tag.
./ecs-build.sh

# 3. Update the ECS task definition with the new tag, then force a redeployment.
aws ecs update-service --cluster app-cluster \
  --service mw-agent-daemon-service \
  --force-new-deployment --region us-east-1

# 4. Wait for the new task to reach RUNNING, then update the debug target IP.
./update-debug-host.sh

# 5. In Cursor / VS Code, press F5 ("Connect to server").
#    You're now in a live debug session — set breakpoints, inspect variables,
#    step through the ECS metrics receiver, etc.
```

## Why This Is Faster

| Before | After |
|---|---|
| Add `log.Println`, rebuild, push, redeploy, read CloudWatch | Set a breakpoint, inspect variables in real time |
| Guess-and-check across multiple deploys | Single deploy, full interactive session |
| Manually find the EC2 host IP after each deployment | `update-debug-host.sh` resolves it automatically |
| Cannot test local collector-contrib patches without publishing | `ecs-build.sh` vendors local packages via `local-deps/` |
| No way to pause or step through startup logic | `stopOnEntry: true` halts at the first instruction |

The net effect is that a cycle that previously took **15-30 minutes** (edit → build → push → deploy → tail logs → repeat) collapses into a **single deployment followed by an interactive debugging session**.
